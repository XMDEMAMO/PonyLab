# 首页终端自动打字补漏与 P1—P8 审计设计

日期：2026-07-23

状态：设计已确认，待实现

范围：修复 P5 首屏 `TerminalTyping` 遗漏，并审计已完成的 P1—P8；不开始 P9。

## 1. 目标与边界

首页阶段一的小终端当前仍停留在 P4 的静态最终态。本次将其补齐为 P5 原计划中的自动打字组件，同时保留现有首屏构图、三阶段滚动状态机、light/dark 双场景图和无 JavaScript 阅读顺序。

本次不会改动背景图片、滚动阈值、阶段二个人名片动画、文章系统、部署配置或依赖；不会引入框架岛、动画库、Pagefind、ClientRouter、播放器或右侧滚动控件。

## 2. 内容与配置

终端以三组“命令 + 输出”循环：

1. `ponylab:~$ whoami` → `Developer · Learner · Dreamer`
2. `ponylab:~$ cat current-focus.txt` → `Astro · TypeScript · Frontend`
3. `ponylab:~$ echo $STATUS` → `记录代码，也记录生活。`

`src/config/home.ts` 将使用类型化终端配置，集中保存：

- `sessions`：可编辑的命令与输出数组；
- `characterIntervalMs`：逐字间隔；
- `outputDelayMs`：命令完成后到输出开始的停顿；
- `sessionHoldMs`：一组内容完整显示后的停顿；
- `transitionMs`：切换下一组前的短淡出时间。

以后替换文案只需修改 `sessions` 中的 `command` 和 `output`。动画组件接口、首页布局和测试选择器保持不变；建议单行控制在约 40 个半角字符以内，过长内容仍允许安全换行而不会造成横向溢出。

## 3. 组件与播放行为

`TerminalTyping.astro` 继续由 Astro 输出静态 HTML，并使用少量原生客户端脚本做渐进增强：

1. 服务端 HTML 默认显示第一组完整内容，确保无 JavaScript 时可读且首屏没有空壳。
2. 浏览器确认允许动画后，视觉层才进入逐字输入；先输入命令，再输入输出。
3. 一组完成后停留，再短暂淡出并切到下一组；第三组后返回第一组循环。
4. 光标在当前输入行末尾闪烁，切换内容时不改变终端外框尺寸。
5. 主题切换只改变现有主题资源和颜色，不重新创建组件、不重置当前会话，也不重复启动计时器。
6. 当前项目没有 ClientRouter；普通页面重新加载时动画从第一组重新开始。若 P10 将来启用 ClientRouter，再按其生命周期统一迁移。

脚本对每个终端实例使用初始化标记，避免开发热更新或重复执行时叠加循环。动画状态以 `data-terminal-state` 和 `data-terminal-session-index` 暴露给自动化测试，但不作为业务 API。

## 4. 可访问性与降级

- 动态视觉文本不反复向读屏器播报；组件提供一份稳定的可访问文本摘要。
- `prefers-reduced-motion: reduce` 下不启动逐字、闪烁或循环切换，直接显示第一组完整内容。
- JavaScript 关闭时同样显示第一组完整内容。
- 页面进入 P5 阶段二时，首屏终端仍跟随现有标题层整体退出；不会在不可见状态继续影响焦点或布局。
- 两行分别保留稳定的最小高度，终端宽度沿用当前响应式规则，避免输入过程产生 CLS 或横向滚动。

## 5. 测试策略

按 TDD 先建立失败测试，再实现：

- unit/contract：终端配置包含三组有效会话和正数时间参数；组件保留静态回退、增强状态标记和 reduced-motion 分支。
- Playwright：观察命令与输出逐字完成、会话索引推进及循环；确认主题切换不重置当前会话。
- Playwright 降级：关闭 JavaScript 与 reduced motion 时第一组完整可见且不进入动画状态。
- 响应式：至少在 320px/390px 检查终端没有横向溢出。
- 回归：重新执行 check、全量 unit、build 和全量 E2E，继续验证 P5 背景不变量和三阶段状态机。

## 6. P1—P8 遗漏审计

终端修复完成后，对 `docs/development-plan.md`、P1—P8 阶段报告、源码、配置、脚本和测试建立逐阶段核对表。每项归为：

- 已实现并有证据；
- 已实现但测试/文档证据不足；
- 按用户决策保留的占位内容或最低素材档；
- 明确属于 P9/P10/P11/P12/P13 的未来工作；
- 已完成阶段中的真实遗漏。

审计结果写入 `docs/stage-reports/p1-p8-implementation-audit.md`。只直接修复边界清楚、不会改变冻结架构的小型遗漏；如发现需要新增依赖、改变部署/路由/内容模型或跨阶段重构的问题，记录证据并停下来交由用户确认。

## 7. 完成标准

- P1—P8 审计有可追踪结果，未来阶段事项不会被误报为遗漏；
- 没有提前引入 P9/P10/P11；
- check、unit、build、E2E 和 `git diff --check` 全部通过。
